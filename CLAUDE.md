## 프로젝트 목적
...
인공지능(ChatGPT, Claude 등)이 출력하는 **Markdown LaTeX 수식**을, **한컴오피스 한글(HWP)의 수식 편집기 문법**으로 변환한다.

LaTeX 문법과 한글 수식 편집기 문법이 서로 호환되지 않아, AI가 만든 수식을 한글 문서에 복사·붙여넣기 하면 깨진다. 이 프로그램은 그 간극을 자동으로 메워, 변환 결과를 한글 수식 편집기에 붙여넣으면 **즉시 올바른 수식으로 렌더링**되도록 한다.

### 변환 예시 (골든 테스트)

이 입출력은 동작의 기준점이다. 변환 로직을 수정할 때 반드시 이 케이스를 통과해야 한다.

**입력 (LaTeX):**
```
$$\lambda(t) = \lambda_0 + A_1 \sin\left(\frac{2\pi}{24}(t - \phi_1)\right) + A_2 \sin\left(\frac{2\pi}{12}(t - \phi_2)\right)$$
```

**출력 (한글 수식):**
```
lambda  (t)= lambda  ` _{`0} +A _{1} sin( {2 pi } over {24} (t`-` phi  _{1} ))+A _{2} sin( {2 pi } over {12} (t`-` phi  _{2} ))
```

## 사용 형태 / 기술 스택

- **형태**: 브라우저에서 동작하는 단일 웹 페이지. 입력창에 LaTeX 수식을 붙여넣으면 변환 결과가 표시되고, "복사" 버튼으로 클립보드에 담아 한글 수식 편집기에 붙여넣는다. **수식 이미지를 업로드/붙여넣기하면 브라우저에서 OCR로 LaTeX를 인식해 같은 변환 흐름에 태운다(아래 "이미지 OCR" 참고).**
- **크롬 확장 프로그램**: `chrome-extension/` 폴더의 내용은 별도의 GitHub 저장소([minigu5/latex-to-hwp-extension](https://github.com/minigu5/latex-to-hwp-extension))에 독립적으로 관리 및 업로드되어야 한다.
- **언어**: JavaScript (프레임워크/빌드 도구 없이 순수 HTML+CSS+JS로 시작. 의존성 최소화).
- **배포**: 정적 파일이므로 파일을 브라우저로 직접 열거나 GitHub Pages 등 정적 호스팅으로 제공 가능.
- **구조 원칙**: 변환 로직(파서/변환기)은 UI와 분리된 순수 함수 모듈로 둔다. 그래야 테스트하기 쉽고, 나중에 CLI나 다른 인터페이스로 재사용할 수 있다.

## 한글(HWP) 수식 편집기 문법 핵심

변환 대상 문법의 기본 규칙. (한글 수식 편집기는 자체 마크업 언어를 쓴다.)

- **공백/띄어쓰기**
  - 백틱 `` ` `` = 좁은 공백(narrow space). 연산자나 기호 주위 간격 미세 조정에 쓰인다.
  - 물결 `~` = 일반 공백.
  - 키워드(`over`, `sqrt`, `sin` 등)는 앞뒤를 공백으로 구분해야 한다.
- **그리스 문자**: 영문 이름 그대로. `\alpha`→`alpha`, `\lambda`→`lambda`, `\pi`→`pi`, `\phi`→`phi`. 뒤에 식별을 위한 공백이 붙는다.
- **분수**: `A over B`. 우선순위를 위해 보통 중괄호로 묶는다 → `{A} over {B}`.
- **첨자**: 아래첨자 `_`, 위첨자 `^`. 여러 글자는 `{}`로 묶는다 → `x_{ij}`, `x^{n}`.
- **루트**: `sqrt {x}`, n제곱근은 `root n of {x}`.
- **함수**: `sin`, `cos`, `tan`, `log`, `ln`, `lim`, `exp` 등 이름 그대로.
- **합/곱/적분**: `sum`, `prod`, `int`. 범위는 첨자로 → `sum _{i=1} ^{n}`.
- **괄호**: 일반 괄호 `(` `)` 그대로 사용. LaTeX의 `\left(` `\right)`는 크기 자동 조절 지시자이므로 변환 시 제거하고 괄호만 남긴다.

## 변환 규칙 (LaTeX → 한글)

> **상세 규칙은 [`CONVERSION_RULES.md`](./CONVERSION_RULES.md)에 정리되어 있다.** 한컴 공식 명세(`한글문서파일형식_수식_revision1.2.pdf`)를 정독해 만든 문서로, 구조 명령·그리스 문자·연산/집합/화살표/기타 기호의 전체 매핑 표와 한글 수식 고유 규칙(공백·항 구분 등), 골든 예시 재해석을 담고 있다. 변환 로직을 만들거나 고칠 때 먼저 이 문서를 참고할 것.

아래는 자주 쓰는 핵심 매핑 요약이다.

| LaTeX | 한글 수식 | 비고 |
|-------|-----------|------|
| `\frac{a}{b}` | `{a} over {b}` | 핵심 변환 |
| `\sqrt{x}` | `sqrt {x}` | |
| `\sqrt[n]{x}` | `root n of {x}` | |
| `x_{i}` / `x_i` | `x_{i}` | 단일 문자도 `{}`로 통일 |
| `x^{n}` / `x^n` | `x^{n}` | |
| `\sum_{i=1}^{n}` | `sum _{i=1} ^{n}` | |
| `\int_a^b` | `int _{a} ^{b}` | |
| `\left(` `\right)` | `LEFT(` `RIGHT)` | 자동 크기 괄호 (명세 권장) |
| `\alpha`, `\lambda`, `\pi`, `\phi` … | `alpha`, `lambda`, `pi`, `phi` … | 그리스 문자 이름 매핑 |
| `\sin`, `\cos`, `\log`, `\lim` … | `sin`, `cos`, `log`, `lim` … | 함수 이름 |
| `\cdot` | `cdot` | |
| `\times` | `times` | |
| `$$ … $$`, `$ … $`, `\[ … \]` | (구분자 제거) | 수식 본문만 변환 |

## 주의사항 및 예외 처리

- **공백 처리는 까다롭다.** 골든 예시의 공백/백틱 배치(예: `lambda  ` 뒤 공백 2개, `` ` _{`0}`` 같은 백틱 위치)는 미묘하다. 모든 공백을 절대 규칙으로 하드코딩하기보다, **골든 테스트를 통과시키는 방향으로 규칙을 점진적으로 다듬는다**. 새 케이스를 발견하면 골든 테스트 모음에 추가한다.
- **한글 수식 문법은 공식 레퍼런스가 빈약**하다. 실제 한글에서 동작 여부로 검증하는 게 가장 확실하다. 확신이 없는 변환 규칙은 추측으로 박지 말고 사용자에게 실제 한글 동작 결과를 물어본다.
- 미지원·미확인 LaTeX 명령을 만나면 임의로 버리지 말고, 변환 불가 항목으로 표시하거나 원문을 보존해 사용자가 알아챌 수 있게 한다.
- **렌더링된 수식 붙여넣기 감지 → 해결 방안 모달:** 사용자가 AI 챗봇에서 렌더링된 수식을 그대로 복사하면 위/아래 첨자 구조가 손실된다. `app.js` 에서는 이를 휴리스틱으로 감지해 경고 모달을 띄운다. 모달은 2단계다 — (1) 경고 화면에서 **"해결 방안 보기"**를 누르면 (2) 번호 탭으로 3가지 해결법을 제시한다: ① LaTeX로 답하도록 요청하는 프롬프트 복사, ② 이미지 캡처(OCR 영역으로 이동), ③ 크롬 확장 프로그램(별도 저장소 `minigu5/latex-to-hwp-extension`). 아울러 ChatGPT, Google AI Studio, Claude(분모-분자 역전 패턴 등)의 평문 복사 포맷을 구분해 최소한의 기호 매핑을 시도하는 Fallback 로직도 함께 제공한다.

## 이미지 OCR (브라우저 로컬)

수식 이미지 → LaTeX 인식까지를 **접속자 브라우저에서** 처리하고, 그 LaTeX를 기존 클라이언트 변환(`window.LatexToHwp.convert`)에 그대로 넘긴다. **서버리스 함수는 쓰지 않는다**(OCR도 변환도 전부 클라이언트). 따라서 `vercel.json`은 정적 서빙 그대로 둔다.

- **모델**: `alephpi/FormulaNet` (PP-FormulaNet-S 파인튜닝, 20M, `image-to-text` ONNX). 라이선스 **AGPL-3.0** — 호스팅 시 소스 공개 의무 유의.
- **런타임**: `@huggingface/transformers`(transformers.js v3)를 CDN ESM으로 로드 → 빌드 도구 없이 동작. 추론은 Web Worker `src/ocr-worker.js`에서 수행.
- **모델 사용법/전처리는 추측 금지**: 작동이 검증된 동일 모델 저자의 구현(Texo-web)을 그대로 따랐다. `VisionEncoderDecoderModel.from_pretrained(..., {dtype:'fp32'})` + 1채널 384×384 텐서를 3채널로 복제해 `model.generate`. 전처리(`app.js`의 `preprocessImage`)는 UniMERNet 규약(그레이스케일·색반전·여백크롭·중앙패딩·mean 0.7931/std 0.1738)을 따른다. 이 값/순서를 바꾸면 정확도가 떨어지므로 함부로 수정하지 말 것.
- **검증**: 헤드리스 Chrome + matplotlib로 렌더한 수식 PNG로 E2E 인식·변환을 확인했다. (KaTeX는 헤드리스에서 웹폰트 글리프가 비어 OCR 테스트 이미지로 부적합 — 실제 렌더 이미지를 쓸 것.)
